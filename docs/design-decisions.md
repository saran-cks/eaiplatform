# Design Decisions (running log)

## DD-1 — Embedding sidecar runs on CPU, not GPU
The model server is **always-on** and serves **one short query per request** at interactive QPS — a *latency* workload, not a *throughput* one. GPU is only justified for bulk throughput, which belongs to the ingestion worker (off-hours, ephemeral GPU). Query sidecar = CPU + (later) ONNX int8. Cheap CPU replicas scale horizontally with QPS.

## DD-2 — bge-m3 backend: FlagEmbedding first, ONNX int8 later
Phase-1 uses FlagEmbedding's reference `BGEM3FlagModel` (PyTorch CPU) because it's correct (dense + learned-sparse guaranteed) and doubles as the **FP32 ground truth** to validate a future ONNX int8 export against. ONNX int8 is deferred until benchmarks prove it's needed. Local CPU bench (12-core box): intra_op=2 → p50 830ms; **intra_op=4 → p50 469ms (sweet spot)**; intra_op=12 → 581ms (regresses). Default set to `EMBED_INTRA_OP_THREADS=4`. Still above the ~250ms target → ONNX int8 is the next lever.

## DD-3 — Embedding/inference stays in Python (not Rust/Go)
The bottleneck is the native ONNX/torch forward pass — identical in any language. A Rust/Go rewrite only replaces the thin request wrapper (microseconds vs ~100s of ms inference) while forcing us to reimplement bge-m3's sparse post-processing. The GIL concern is moot: inference releases the GIL, so the thread-pool offload already gives true parallelism. Revisit only at very high QPS for p99.

## DD-4 — Messy user queries: rewrite-before-embed, not embed-as-is
bge-m3 handles typos/casing/punctuation/multilingual for free (do **not** lowercase/scrub — it's cased; over-cleaning strips signal). But it can't condense rambling or resolve cross-turn references ("why is *it* failing?"). So a **cheap LLM (Haiku) query-rewrite** turns the messy/multi-turn input into a clean **standalone** search query, and *that* gets embedded. Two distinct texts:
- **rewritten query → embedding/retrieval** (clean, intent-focused, context-resolved)
- **raw query → answer LLM** (preserve the user's actual words/tone)

Skip the rewrite when it can't help (first turn + already-clean/standalone query); only rewrite on multi-turn or long/rambling input.

## DD-5 — Two-tier response cache
- **L1 — raw-normalized key**, checked **first**, before any LLM call. Instant/free; catches exact repeats.
- **L2 — rewritten-query key**, checked only after L1 miss (rewrite already paid to embed). Catches different phrasings of the same intent; shareable across users with the same tenant + scope because the rewrite is standalone.

Pay the rewrite only on an L1 miss. Fill L2 only with the **standalone** rewritten form (never anything carrying session-specific references), or the next user's context gets a wrong answer.

## DD-6 — Agent semantic retrieval reuses the SAME embedding sidecar
Endpoint separation (chat vs agent) is about *orchestration flow*, not about avoiding embeddings. If an agent needs **semantic** Qdrant data it MUST embed via the same sidecar (`RetrieverPort`), exposed as a scope-checked agent **tool** that reuses the chat retrieval use-case — never a second embedding model. Only **exact/filter lookups** (by `tenant_id`/`permissions`/`ticket_id`/`doc_id`) skip embedding (Qdrant payload filter / Postgres). The real-time-status agent (1a) bypasses Qdrant entirely and hits live systems via MCP. **Not built yet — implement when agent tools land** (also noted in the agent flow section of `core-api-architecture.md`).

## DD-7 — Agent security: contain at the ACTION layer, not the input layer
**Thesis:** Prompt Guard (the sidecar) is only the **outer perimeter**. It catches blatant one-shot jailbreaks and misses what matters for an agent wired to prod systems (ServiceNow, GitHub, MCP tools): **indirect injection** (malicious instructions inside a ticket body / PR comment / retrieved doc / tool response) and **slow, gradual context poisoning** across turns. No input classifier can close that gap. So we **assume the agent's context will be poisoned** and move the trust boundary to the **action/authorization layer**. The question is never "is this prompt malicious?" but "is **this agent**, with **this scope**, allowed to take **this action** on **this system**, right now — and is it reversible?" The LLM **proposes**; deterministic code **decides**. Security guarantees live in the code *around* the model, so "the agent got talked into it" is never sufficient to cause harm.

**Layers (defense-in-depth):**
1. **Least-privilege scoped tools** *(foundation, partly built — `PermissionScope` gates MCP tool registration)*: per-task minimal tool set, read-only by default; writes to prod systems are a separate elevated class.
2. **Deterministic action-policy gate** *(key piece to add)*: an OPA/Cedar-style allow-list keyed on scope + action-class + target sits between the agent's *proposed* tool call and execution. A fully manipulated agent still hits the wall ("this scope may never delete prod resources"). Highest-leverage control.
3. **Human-in-the-loop for irreversible / high-blast-radius actions** *(already FUTURE in architecture)*: plan-then-apply; no merge/deploy/destructive change without explicit approval.
4. **Untrusted-data handling**: all tool/retrieved content is **data, never instructions**; structural separation of system vs user vs tool content; **provenance/taint** so high-taint external content can't authorize high-privilege actions. Second use of the guard: screen **tool outputs / retrieved docs** for indirect injection, not just the user prompt.
5. **Trajectory monitoring** *(for the slow attack)*: because the gate is at action-time, gradual poisoning still hits the policy wall when it finally acts; on top, an **independent monitor** watches the *sequence* of tool calls for drift toward sensitive ops + **cumulative session risk** (not per-message). Maps onto the Phoenix observability/drift/eval layer.
6. **Blast-radius containment (assume compromise)**: sandboxed code exec (E2B, FUTURE), **short-lived per-action scoped credentials** (agent never holds standing prod keys), rate limits / circuit breakers on sensitive tools, `agent_reaper` kills runaway/orphaned agents; prefer reversible, idempotent actions.
7. **Full audit**: every tool call + policy decision emits an OTel span to Phoenix (forensics + eval datasets).

**Built vs gap:** *Already designed* — `PermissionScope` top-down, MCP tool gating, scope-checked tool registration, human-in-the-loop (FUTURE), `agent_reaper`, observability/drift/evals, E2B sandbox (FUTURE). *To add explicitly* — the deterministic **action-policy engine**, **provenance/taint**, **tool-output guard screening**, **trajectory monitor + cumulative risk**, **per-action short-lived scoped credentials** + read/write tool separation + approval workflow. **Implement when agent tools land** (see DD-6 and the agent flow in `core-api-architecture.md`). → Pressure-tested and formalized into concrete, enforcement-checkable decisions in **DD-8…DD-12**.

## DD-8 — Policy Decision Point (PDP): a mandatory, agent-adversarial action chokepoint
**Refines DD-7.** Every agent-initiated effect on an external system (MCP tool call, sub-agent delegation, artifact-as-action) routes through a single deterministic Policy Decision Point *before* execution. The PDP is **default-deny** and must be correct under the assumption that **the agent is the attacker** — its decision may not depend on the model being honest, nor on taint (see DD-9). The LLM *proposes*; the PDP *decides*. "The agent was convinced" can never authorize an effect, because conviction lives in the data plane and authority lives in the control plane (DD-7), and they do not touch.

Pressure-tested amendments (vs. the naïve "allow/deny on action-class"):
- **Gate flows, not just actions.** Exfiltration is two individually-allowed actions (read in-scope source → write in-scope sink). The PDP evaluates `(scope, action-class, target, source-of-data, reversibility, quantitative bound)` — read-source × write-sink pairs are policy objects, not free. *Two-plane invariant, amended:* the model may move data **only within a policy-approved flow**.
- **Bounded parameters, not coarse classes.** `write:servicenow:ticket` cannot distinguish "close one ticket" from "close every P1." Prod-write tools MUST be individually modeled with typed, bounded parameters (max count / rate / blast-radius). This is an accepted, non-zero modeling cost — coarse classes are gameable on parameters.
- **Canonical targets.** The PDP resolves the real target itself (adapter-bound concrete resource); it never trusts a model-supplied label like `target=dev-db`.
- **Delegation attenuates.** The PDP sits at the execution boundary for *all* agents incl. sub-agents / A2A peers; scope may only *narrow* across delegation, never widen (closes the confused-deputy bypass — A2A is already stubbed in `src/adapters/agent/a2a/`).

**Why highest-leverage:** a fully poisoned agent still hits a deterministic wall it cannot argue with. This is the single control that converts "we have a security philosophy" into "an agent physically cannot mutate prod from poisoned context."
**Enforcement check (future):** no code path may invoke `MCPConnectorPort.call_tool` (or any write-capable tool/adapter) except via the PDP; a write tool with no registered bounded-parameter policy is default-denied; an architecture-style test asserts the chokepoint has no bypass. → **Implemented** (core, no caller yet): `core/use_cases/policy/policy_decision_point.py` (default-deny `decide()` — scope, attenuating delegation, canonical-target spoof check, flow gating, bounded params, reversibility/approval, taint-as-signal); `src/tests/test_pdp_chokepoint.py` (static no-bypass guard) + `test_policy_decision_point.py`. **First caller now wired** — the PDP-guarded MCP connector (DD-14, `adapters/mcp/connector.py`) routes every tool call through `decide()`; the chokepoint allowlist is no longer empty. Still future: the credential broker (DD-10), approval backend (DD-12).

## DD-9 — Taint/provenance is a SIGNAL, not a gate (you cannot trace influence through an LLM)
Every context fragment is tagged with source-trust (user / first-party retrieval / external tool output / external doc). **But taint must never be the thing a hard deny depends on.** Pressure test: you cannot prove whether tainted content *influenced* an LLM decision, and if you treat all model output as tainted (its context is tainted), taint becomes universal and the PDP denies everything → useless. Therefore taint tracks only **provable data-flow into action parameters**, and is consumed as: (a) a risk input to the trajectory monitor (DD-11), (b) friction/justification-raising on high-privilege actions, (c) audit/forensics. It is **defense-in-depth, not load-bearing.** DD-8 must be correct with taint unknown.
**Enforcement check (future):** removing the taint subsystem must not make any DD-8 deny become an allow (i.e., no hard policy rule keys on taint alone).

## DD-10 — Capability-based, short-lived per-action credentials (assume compromise)
The agent holds **no standing prod credentials.** To act, it requests a short-lived, single-purpose, scope-narrowed capability token minted per approved action. This caps blast radius when (not if) an agent is compromised, and makes every credential issuance an auditable PDP event. Prefer reversible, idempotent operations.
**Enforcement check (future):** no long-lived prod secret is reachable from the agent runtime; credential TTL ≤ action window; every mint is tied to a PDP allow decision.

## DD-11 — Trajectory monitor + cumulative session risk (the answer to the SLOW attack)
**Promoted from "hardening" to required.** Pressure test: the user's actual threat (slow context poisoning / gradual privilege elevation) is a chain of *individually-allowed* actions, which a stateless per-action PDP (DD-8) cannot catch. An **independent monitor** scores the *sequence* of tool calls and accumulates **session-level risk** (not per-message), watching for drift toward sensitive ops, read-then-exfiltrate shapes, and elevation gradients; crossing a threshold throttles, forces re-approval (DD-12), or kills the session (`agent_reaper`). Rides the Phoenix observability/eval layer. **DD-8 stops the single catastrophic action; DD-11 stops the slow accumulation — neither alone is sufficient.**
**Enforcement check (future):** a synthetic "thousand small cuts" trajectory must trip the monitor even though every individual action passes DD-8. → **Implemented** (core, no caller yet): `core/use_cases/policy/trajectory_monitor.py` (stateful per-session cumulative risk; signals: mutating-drift, read-then-exfiltrate, elevation-gradient, privilege-growth, probing; OK→THROTTLE→REQUIRE_APPROVAL→KILL). `test_trajectory_monitor.py::test_thousand_small_cuts_trips_monitor_though_each_passes_pdp` runs every action through the real PDP (asserting ALLOW) and shows the monitor still escalates to KILL. **First caller now wired** — the MCP connector (DD-14) calls `observe()` after every PDP decision and enforces the verdict (KILL → `TrajectoryKill`, REQUIRE_APPROVAL → refuse), and the **agent runtime now drives that connector** (DD-15), so the loop is closed end-to-end: a `TrajectoryKill` records the session in `AgentKillRegistry`, surfaces a `killed` event, sets `AgentStatus.KILLED`, and the **`agent_reaper` daemon force-terminates it** as a backstop. Still future: Redis-backed `SessionRiskStore` (in-memory now), Phoenix spans.

## DD-12 — Reversibility (not "sensitivity") is the axis for human-in-the-loop
Approval gating keys on **irreversibility / blast-radius**, not a vague "sensitive" label. Plan-then-apply: no merge/deploy/destructive/irreversible action executes without explicit human approval; reversible idempotent actions flow without friction. Reversibility is a first-class attribute of every tool's policy contract (consumed by DD-8 and DD-11).
**Enforcement check (future):** every write-capable tool declares a reversibility class; anything `irreversible` with no approval token is default-denied at the PDP.

### Pressure-test summary (why these five, in this shape)
Red-teamed before adoption. Survivors and corrections: **two-plane separation holds** but the invariant became "data moves only within an approved *flow*"; **DD-8 holds and hardened** (flows + bounded params + canonical targets + attenuating delegation, correct under an adversarial agent); **DD-9 demoted** (influence is unprovable through an LLM → taint is a signal, not a gate); **DD-11 promoted** (the per-action gate is blind to slow accumulation by construction). The honest one-liner: **DD-8 stops the single catastrophic action; DD-11 stops the slow accumulation; DD-10/DD-12 limit blast radius; DD-9 is a smoke detector, not a lock.** All five carry an *Enforcement check* so we can later audit designed-vs-enforced (cf. the architecture-boundary test precedent).

## DD-13 — Untrusted retrieved-content handling for RAG (chat path)
**Refines DD-9, applied to RAG reads.** Retrieved corpus chunks are **untrusted data, never instructions**. Indirect injection (a malicious instruction embedded in a corpus doc) is mitigated in layers; *scanning is the secondary layer, structure is the primary one.*

- **Layer 0 — Structural separation in the system prompt (primary, free, chat path). → DONE.** Context is injected in a delimited, **datamarked** block; the system prompt declares it untrusted reference data — "quote it, never obey it." This is the actual gate against indirect injection and costs nothing per query. **Implemented** in `send_message.py` (`_format_context` + `_neutralize_delimiters`): each context line is prefixed with a sentinel (`│ `) the prompt tells the model to trust, and any chunk text forging the begin/end markers (or a bare dashed rule) is neutralized **before** rendering — so a passage cannot close the block and issue instructions. A delimiter you can spoof isn't structure; neutralization removes the spoof, datamarking gives a positive "this is data" signal. Verified by `test_chat_security.py` (forged END marker leaves exactly one real closing marker; injection survives only *as datamarked data*).
- **Layer 1 — Scan at INGESTION, not retrieval (amortized).** The **ingestion worker** (`ingestion_worker/`, not yet built — this is a build requirement for it) screens each chunk **once at ingest**: cheap heuristics first (imperative 2nd-person directives, "system prompt"/"ignore previous" language, zero-width/invisible unicode, HTML comments, base64 blobs, embedded delimiters), then the Prompt Guard (PG2) classifier only on suspects. Result is persisted as **first-class fields on the chunk payload** (Qdrant), alongside `permissions`: e.g. `screened: bool`, `injection_risk: float`, `provenance/source-trust`. Cost is paid once per document, off the hot path — never once-per-query-per-popular-doc.
- **Layer 2 — At retrieval, consume the stored flag (cheap, no hot-path model call).** Query time reads the precomputed flag. A flagged chunk is **neutralized, not hard-dropped**: down-ranked, wrapped in explicit "⚠ untrusted/flagged" markers, taint raised. Hard-drop is itself a weapon — poisoning one sentence of a legit doc would erase it from retrieval (a DoS on knowledge). Hard blocking is reserved for the **agent** path (where a chunk could authorize an action, DD-8); in chat the blast radius is only "wrong answer," so over-blocking costs more than it protects.
- **Layer 3 — Optional batched retrieval-time classifier pass** for high-risk tenants only. Opt-in, not default.

**Design rules:** *ingest-time > retrieval-time* (scan once, amortize, keep the hot path clean); *neutralize/flag > block* for chat; the classifier score is a **signal feeding the taint flag, never an autonomous gate** (consistent with DD-9 — structure is the gate, the score is the smoke detector).
**Scope note (chat vs agent):** in chat (read-only, pre-scoped retrieval) a successful injection is bounded to *answer integrity / misinformation*, not breach — confidentiality/isolation already holds by construction. DD-13 closes the answer-integrity gap; it does not exist to prevent a breach (there isn't one to prevent on the chat path).
**Enforcement check (future):** every chunk written by the ingestion worker carries a `screened` flag + `injection_risk` (no chunk reaches Qdrant unscreened); the chat system prompt contains the untrusted-context framing; a synthetic poisoned chunk is down-ranked/marked at retrieval rather than silently used; removing Layer 1/3 must not change chat *confidentiality* (only answer integrity).

## DD-14 — The MCP connector IS the PDP chokepoint (DD-8/DD-11's first caller)
**Implements DD-8 + DD-11.** The PDP-guarded MCP connector (`adapters/mcp/connector.py`) is the single, allowlisted place an external tool is executed. Every `call_tool` runs the same gauntlet *before* the transport is touched: `PDP.decide()` → `TrajectoryMonitor.observe()` (fed on **every** decision, including denies, so probing accrues risk) → enforce **KILL > deny > require-approval > proceed**. A `ToolSpec` (one per read-only tool) is the single declaration that produces *both* the `list_tools` view and the PDP `ToolPolicy`, so the two can't drift; a tool with no spec has no policy and is default-denied. The raw transport (`MockMCPTransport` now; real `ClientSession` later) does **no** authorization — it already happened at the chokepoint.

Small decisions taken here: **(a)** `MCPConnectorPort.call_tool` gained an optional `session_id` so DD-11 can key cumulative risk per agent-session (falls back to subject/tenant); **(b)** phase 1 tools are **read-only** (reads may target any environment incl. prod — blast radius is "wrong answer"; writes are FUTURE and will narrow `allowed_environments` + carry bounded params); **(c)** **no HTTP route yet** — the agent runtime is the intended caller, so the chokepoint allowlist stays exactly `{connector}` rather than also trusting a route.
**Enforcement check (real now):** `test_pdp_chokepoint.py` allowlist = `{adapters/mcp/connector.py}` — any other module invoking `.call_tool(` without the PDP fails the build; `test_mcp.py` proves deny-before-transport and that a KILL trajectory vetoes even a PDP-allowed action. Live: smoke-tests **ST-3**.

## DD-15 — The agent runtime is the chokepoint's runtime caller (and KILL gets reaped)
**Closes DD-8/DD-11 end-to-end.** DD-14 built the chokepoint; this makes it *load-bearing* by routing real traffic through it. The `LangGraphRunner` workers fetch external data via `MCPConnectorPort.call_tool(... session_id=agent_session_id ...)` — so per-action policy (DD-8) and cumulative session risk (DD-11) actually run on every agent tool call, keyed per agent-session. Failure semantics are deliberate: a **PDP deny degrades one worker** (the run continues with that source missing — denial is not a crash), but a **`TrajectoryKill` tears the whole session down** (cumulative risk is fatal to the session, not to a single source). On KILL the runner emits a `killed` event, records the session in `AgentKillRegistry`, `RunAgentUseCase` sets `AgentStatus.KILLED`, and the `agent_reaper` daemon drains the registry and force-terminates as a backstop.

One decision taken here: routing the runner through the connector adds a second `.call_tool(` site, so the static chokepoint guard moves to a **two-tier allowlist** — `connector.py` (calls the raw transport, PDP-first) **+** `langgraph_runner.py` (calls the *connector port*, which IS the PDP entry). This is safe because DI only ever hands the runner the guarded `MCPConnectorPort`, never the raw transport, so it structurally cannot bypass; the guard still fails the build on any *new, un-reviewed* `.call_tool(` site.
**Enforcement check (real now):** `test_agent_mcp_wiring.py` — in-scope workers reach the transport, an under-scoped worker is PDP-denied while the run survives, a KILL emits `killed` + records the session + never hits the transport, `RunAgentUseCase` persists `KILLED`, and the reaper terminates drained sessions. Still future: `api/routes/mcp.py`, real `ClientSession` transport (ST-3), DD-10 broker / DD-12 approval.

## DD-16 — Cumulative session risk persists in a shared backend (DD-11 must survive scale)
**Hardens DD-11.** The trajectory monitor's in-process risk is correct for one worker but **defeated by horizontal scale** (risk doesn't add up across processes) and lost on restart — the slow attack could just round-robin its calls across replicas. So `SessionRiskState` is persisted to Valkey via a `SessionRiskStorePort` (`risk:{session_id}`, TTL = agent-session window), and the monitor hydrates/writes-back per action (`observe_async`). Two rules: **(a) fail-soft** — a backend outage degrades to in-process accumulation (a weaker control) but **never** breaks the action path or *bypasses* the gate (in-process still accrues); **(b)** load→modify→save is **not atomic** — acceptable because within one agent session tool calls are sequential (one worker at a time); a Lua CAS / atomic-increment is FUTURE for genuinely concurrent same-session writers. Off-switch `RISK_STORE_ENABLED` for single-worker dev.
**Enforcement check (real now):** `test_session_risk_store.py` — two monitor instances sharing one backend keep accumulating for the same session (cross-worker), distinct sessions don't bleed, and a failing cache still returns a verdict + accrues in-process. Built on the existing `CachePort` (no second client; no adapter→adapter import).

**Addendum (2026-06-30) — rule (b) was wrong, and the gap was reachable: load→score→save now runs under a per-session lock.** The original claim "within one agent session tool calls are sequential (one worker at a time)" is **false for this codebase**: the agent runtime (`langgraph_runner.py`) is a Map-Reduce fan-out — `_route_planner` emits parallel `Send`s and `code_worker` + `ticket_worker` both drive the *same* connector with the *same* `session_id` on the *one* shared `TrajectoryMonitor`. The store path makes `observe_async` span `await` points (load → score → save), so two concurrent calls interleave and **lose an update** — and a lost update always *under-counts* risk, the single direction that weakens KILL / REQUIRE_APPROVAL detection (an attacker who fans mutating calls out in parallel suppresses cumulative risk: an exploit of the very mechanism DD-11 exists for). Note the irony: the pure in-process `observe` was already safe (fully synchronous, no interleave); **persistence introduced the race**, and `self._sessions[sid] = loaded` could even clobber good in-process state with a stale snapshot. Fix: `TrajectoryMonitor.observe_async` holds a **per-session `asyncio.Lock`** across the whole read-modify-write (per-session, so distinct sessions don't serialize). Atomicity is enforced one level up in the use-case, where the multi-step invariant lives — *not* by reimplementing the stateful `_score` in a Lua script (that would duplicate the most security-critical logic in a second language and invite drift). A single agent session's SSE run is owned by one worker, so a per-*process* lock covers the reachable concurrency; genuinely concurrent same-session writers spread *across processes* remain FUTURE — and the right tool there is a Valkey **CAS** (optimistic version check around the existing Python `_score`), **not** a distributed lock (no fencing token; mutual exclusion isn't what's needed — atomic compare-and-set is). If/when that lands, a conflict must fail **safe** (keep the higher contending risk), never last-writer-wins downward.
**Enforcement check (real now):** `test_session_risk_store.py::test_concurrent_same_session_does_not_lose_updates` — a seeded session driven by two `asyncio.gather`'d `observe_async` calls over a control-yielding cache double must total the same as the strictly-sequential baseline (verified to fail — 0.48 vs 0.72 — with the lock removed); `test_distinct_sessions_do_not_serialize` holds the per-session granularity.

## DD-17 — Observability is a vendor-neutral port; Phoenix is just one adapter
**Full session observability, swappable by construction.** Tracing/evals/datasets/drift all flow through one neutral `ObservabilityPort` (`core/ports/observability.py`): a `SpanKind` taxonomy, an `ObsAttr` attribute vocabulary, an `ObsSpan` enrichment handle, and read/write methods. `core` and use-cases speak **only** this vocabulary — no OpenInference/OTel/Phoenix symbol appears in `core` (enforced by `test_architecture.py`: `observability/` and the adapter may import OpenInference; `core` may not). Switching to **Langfuse** is therefore: add `adapters/observability/langfuse/` mapping neutral→Langfuse and rebind `Container.observability`. Nothing else changes.

The production adapter (`adapters/observability/phoenix/`) is deliberately **lightweight**: `arize-phoenix-client` + `openinference-semantic-conventions` + `openinference-instrumentation` only — **not** the heavy `arize-phoenix` package. Reason: the Phoenix *server* (the Docker container) is what computes UMAP / embedding-drift / the UI; the pip package's only unique trick is `px.launch_app()` (an *embedded* server), which would duplicate the container. We unlock UMAP/drift the server-appropriate way — **attach embedding vectors to spans** (OpenInference embedding conventions) and compute a programmatic drift signal ourselves (centroid Euclidean/cosine + PSI, the methodology Arize documents). No pandas in the always-on API.

Decisions taken here: **(a)** spans set OpenInference attributes on *plain OTel spans* over our existing `TracerProvider` — no `phoenix.otel.register()` (it would hijack the global provider); **(b)** **fail-soft is absolute** — a Phoenix/exporter/client outage degrades to no-op spans + empty reads, never an error on the request path (observability must not be able to break chat/agents); **(c)** **evals reuse our `LLMPort`** (Bedrock) as the judge, mirroring Phoenix's pre-tested templates (Hallucination/QA Correctness/Relevance/Toxicity, single-word rails, temp 0), logged back as span annotations — avoids a second LLM client *and* `arize-phoenix-evals`/pandas; **(d)** auto-instrumentation (Bedrock + LangChain/LangGraph) is an **opt-in** toggle on top of the explicit, port-driven domain spans, which remain the load-bearing signal; **(e)** Phoenix **Sessions** is the spine of "session observability" — every span carries `session.id`, so a whole chat/agent run groups into one session view.
**Enforcement check (real now):** `test_observability.py` — neutral→OpenInference mapping (+ asserts our literal keys match the installed constants), pure drift math + the Valkey drift tracker (warm-up→baseline→drift, fail-soft), the NoOp adapter, Phoenix span emission via an in-memory OTel exporter (OpenInference attrs + session id + mid-span enrichment + drift feed), the LLM-judge panel, and the connector/chat producer spans. `test_architecture.py` keeps OpenInference out of `core`. Live: smoke-tests **ST-4**.

## DD-18 — Production inference runtime: ONNX Runtime for both CPU sidecars, but int8 only where accuracy is verified
**Both sidecars are CPU-only services (no GPU by design) on the request-blocking path**, so the prod target is **ONNX Runtime**, not PyTorch eager — the headline win is **dropping torch entirely** (≈1GB) for onnxruntime (≈200MB): smaller images, lower RAM, faster cold-start/scale-out, cheaper instances. (That same torch footprint is exactly what made the prompt_guard container ENOMEM on the dev box — Session 3 / **ST-5**.) Precision is decided **per model by where it sits**, not uniformly:

- **(a) Embedding sidecar (bge-m3) → ONNX int8.** It runs on *every* chat/search query pre-LLM and is the dominant CPU latency (FP32 ≈469ms p50; int8 target ≈250ms), so int8's ~2–3× speedup + ~4× memory cut is worth it. **Gate:** int8 must be validated against the FP32 reference (`backend="flag"`) — dense cosine-sim and retrieval recall on a sample set (`scripts/export_quantize.py` + `EXPORT_RUNBOOK.md`; the sparse head reuses the shared `sparse_weights_to_dict` so FP32 and ONNX can't drift). The `EMBED_BACKEND=flag|onnx` switch already exists; FP32 stays the correctness ground-truth and the fallback.
- **(b) Prompt guard (PG2 86M) → ONNX, but FP32/FP16 first, int8 only after re-validation.** The win here is the **lighter no-torch image/RAM**, not speed (86M is already fast). Because it's a **security control with a 0.5 decision boundary**, int8 quantization could shift the boundary into **false negatives (injections passing)** — unacceptable silently. So the default is ONNX FP32/FP16 (operational win, zero accuracy risk); int8 is allowed **only** after re-measuring detection rate + re-tuning `GUARD_THRESHOLD` against a labeled set. Safety beats a few ms.

Net: **no torch in prod** (one inference runtime across both sidecars), with int8 applied to the knowledge path (measured by recall) and deliberately *withheld* from the safety path until proven. Cost acknowledged: each model needs an export/quantize step — the bge-m3 export is the RAM-heavy one (memory `embedding-sidecar-onnx-pending`); the guard export is comparatively trivial.
**Enforcement check (gates, not yet a unit test):** embedder int8 ships only when the export-validation diff vs FP32 passes (recall/cosine thresholds in `EXPORT_RUNBOOK.md`) and re-benched (**ST-5**-style); guard int8 ships only behind a recorded detection-rate re-validation + threshold re-tune. Until then: embedder FP32 (`flag`) and guard FP32 remain the bound defaults. A `test_embedder` parity check (ONNX vs FP32 within tolerance) becomes the durable guard once the int8 artifact exists.

## DD-19 — Frontend is a static React/Vite SPA (the 5th deployable); auth has a dev-mint path and a prod Cognito path
**The platform gets a UI — a new, fifth deployable (`frontend/`) that, like the other four, shares no code and is coupled only over the wire (the Core API's HTTP/SSE surface).** It is decided as a **static React + Vite + TypeScript SPA**, not SSR/Next.js. Rationale: every surface lives **behind auth** and talks to a **pure JSON/SSE API** — there is no SEO, no public content, no server-rendered data need — so SSR's only real payoff (first paint of public pages) does not apply, while it would add a Node runtime to operate and secure. A static SPA builds to assets served by a CDN / the API's static mount: the lowest-maintenance shape for an internal authed tool. The Core API stays the single backend; the SPA is a client, never a BFF.

**Locked library set (deliberately small, all mature, all tree-shakeable):** Vite + React + TS · **TanStack Router** (type-safe, first-class protected-route layouts) · **TanStack Query** (server-state cache/retries) · **`@microsoft/fetch-event-source`** for SSE · **shadcn/ui** (Radix + Tailwind) for components · **Zustand** for the small amount of client state (auth/session). shadcn was chosen over a batteries-included kit (MUI/Mantine) specifically to **own the component source** — no vendor upgrade treadmill, lowest long-term maintenance — accepting Tailwind as the cost.

**Why `fetch-event-source` and not native `EventSource` (load-bearing):** the two streaming surfaces are **POST + SSE**, and the browser's native `EventSource` is **GET-only**. So `fetch-event-source` is mandatory, not a preference. The streams are also shaped differently and the client must handle both: **chat** emits bare `data: <token>` frames terminated by `data: [DONE]` (+ an `event: error` frame on failure); **agent** emits *named* events — `thought` / `worker_start` / `worker_done` / `synthesis` / `output` / `error` / `done` — each with a JSON `data` payload. Agent runs also produce **Monaco-ready code artifacts** (`/agent/{id}/artifacts`, fields `name`/`content`/`language`), so the agent surface carries a Monaco editor dependency the chat surface does not.

**Auth — two paths, plan for both (prod is not yet wired e2e):**
- **Token model (already true server-side):** the Core API **only verifies** a JWT (HS256 shared secret) and builds `PermissionScope` from claims `tenant_id`, `permissions[]`, `sub` (`api/middleware/auth.py` → `PermissionScope.from_claims`). **There is no login/token-issuing route in the Core API and none is planned** — the token is minted *outside* it. The SPA never "logs in against the backend"; it acquires a token, then sends `Authorization: Bearer <jwt>` on every call.
- **Dev path (build now):** a local **dev login page** that accepts an externally-minted HS256 token (paste a token, or a dev-only helper that mints one from the shared secret with chosen `tenant_id`/`permissions`/`sub`) and stores it for the session. Unblocks all UI work with no IdP.
- **Prod path (design now, wire later):** **AWS Cognito** via OIDC — the user authenticates with Cognito, and the **Cognito profile/group attributes map to the `permissions`/`tenant_id` claims** that become the `PermissionScope`. The SPA runs the standard OIDC Authorization-Code-+-PKCE redirect, receives the JWT, and uses it identically to the dev token. **E2E prod auth (Cognito user pool, claim mapping, and the backend's RS256/JWKS verifier swap) is NOT in place yet** — so the SPA's auth layer is built behind a single `AuthProvider` seam: dev-mint and Cognito-OIDC are two adapters of one interface, swappable by config, so no feature surface changes when prod auth lands. (Mirrors the backend's own "swap the verifier adapter for RS256/JWKS later" note.)
- **Permission-scoped rendering is defense-in-depth, never the gate:** the SPA reads `permissions[]` from the token to decide which surfaces/actions to *show*, but this is purely UX — the backend re-enforces every scope (Qdrant payload filter, PDP, route checks). A hidden button is not a security control; the server is. (Consistent with the platform's "the LLM proposes, deterministic code decides" stance.)

**Known API gaps the FE plan routes around (does not block on):** (a) the **chat stream returns bare tokens, no structured citations** — a "sources" panel either calls `/search` alongside or waits for a future citations SSE event/field (flagged, not built); (b) **`/dashboard` (SSE drilldown) is not built yet** server-side, so the dashboard surface is planned but its data source is pending that route + the ML pipeline; (c) observability reads (`/observability/{traces,evals,datasets,drift}` + `/feedback`) **are** live and can back an admin/ops surface today.

**Enforcement check (future, once code lands):** the SPA never calls a backend auth/login endpoint (there is none); every API call carries `Authorization: Bearer`; the dev-mint and Cognito auth providers satisfy one `AuthProvider` interface (swap = config only); SSE is consumed via `fetch-event-source` (POST), handling both the bare-token chat stream and the named-event agent stream. Visual design (palette, typography, spacing, theming) is tracked in `docs/frontend-build-plan.md` and is **not** part of this decision.

**Addendum (2026-06-29) — the Observability surface is a permission-gated Phoenix launcher, not a native re-render of the obs feeds.** Phoenix already ships a mature UI (trace explorer, evals, datasets, drift charts) at its own origin; reimplementing those four feeds (`/observability/{traces,evals,datasets,drift}`) natively in the SPA would be strictly worse and largely redundant. So: the Observability tab is a single **"Open Phoenix ↗"** launch button (opens the configured Phoenix URL, e.g. `localhost:6006`, in a new tab); native scoped trace/eval/dataset/drift views are **deferred (FUTURE EXTENSION)** until a multi-tenant prod deploy needs tenant admins to see *their* slice without raw Phoenix access. Non-dev users don't see the button. **Addendum (2026-07-01) — the backend Cognito verifier landed behind a `TokenVerifierPort` seam; verify-only is reaffirmed.** DD-19's prod-auth plan ("swap the verifier adapter for RS256/JWKS later") is now implemented. Because the HS256 decode had lived *inline* in `api/middleware/auth.py`, there was no adapter to swap — so the seam was extracted first: **`core/ports/token_verifier.py`** (`TokenVerifierPort.verify(token) -> Mapping`, raising `TokenVerificationError`), with two adapters under `adapters/auth/` — `HS256TokenVerifier` (dev) and `CognitoJwtVerifier` (prod, RS256 + JWKS) — bound in `config/di.py` (the sole wiring point) by a new `AUTH_PROVIDER` setting. The middleware is now transport-agnostic (`await verifier.verify()`), and the 401-on-bad-token / 403-on-bad-claims split is unchanged.

Three sub-decisions worth recording: **(1) Verify-only is reaffirmed, not widened.** When asked for backend `/auth` login/refresh/logout endpoints, we declined: DD-19's "no login/token-issuing route" invariant holds. The frontend Option-B adapter is separate-deployable work (`frontend/`), not the Core API.

## DD-20 — The Qdrant+Postgres dual-write is not transactional: Qdrant is the system of record, write-order makes the gap self-healing, the outbox is for deletion-completeness
**Names a known distributed-systems anti-pattern and states why the current shape is safe for retrieval — and where it genuinely isn't.** The ingestion worker writes each changed chunk to **two** stores with no shared transaction: vectors → **Qdrant** (the retrieval index core-api reads) and bookkeeping rows → **Postgres** (the `chunk_registry` that powers delta/dedup). A crash or partial failure between the two leaves them inconsistent — the textbook dual-write hazard (Kleppmann's own search-index-beside-a-database example). We do **not** reach for a distributed transaction / 2PC across Qdrant+Postgres (operationally heavy, and neither store gives us a usable cross-store atomic commit). Instead three properties bound the damage:

- **Qdrant is the de-facto system of record for retrieval; the registry is a *derived*, rebuildable index.** Core-api never reads the registry — only Qdrant. So a registry that disagrees with Qdrant cannot, by itself, return a wrong answer to a user.
- **Write-order is the load-bearing safety property: Qdrant FIRST, registry SECOND** (`orchestrator.py`). This makes the registry only ever **lag** Qdrant, never **lead** it. The two failure directions are asymmetric:
  - *Registry lags* (Qdrant ahead): the next ingest's `diff()` sees the chunk missing from prior-hashes → re-embeds + re-upserts (idempotent by `chunk_id`) → re-records. **Self-heals; cost is redundant re-work, never a retrieval gap.** `diff()` *is* the reconciliation loop for any re-ingested doc.
  - *Registry leads* (Qdrant behind): `diff()` would mark a chunk "unchanged" and skip a chunk that isn't actually in the index → **silent retrieval loss.** The write-order **structurally excludes** this direction.
- **`chunk_id` idempotency** (`sha256(source⟂native_id⟂field_role⟂seq)`, `contracts/chunk_identity.md`) makes every retry a no-op, not a duplicate.

**The residual risk — and the real justification for the outbox.** The self-heal only fires *if the document is re-ingested.* A partial upsert (Qdrant ✓, registry ✗) on a doc that is **never touched again** leaves chunks in Qdrant the registry doesn't track. Retrieval stays correct, but a **registry-driven bulk delete** (tenant offboarding, GDPR erasure) walks the registry and would **miss those untracked Qdrant points** — a deletion-**completeness**/compliance gap, not a retrieval gap. This — not retrieval correctness — is the primary reason the **durable outbox / reconciliation** is still required (build-plan Fork #4, Phase-3): an outbox that records intent before either write and a sweeper that reconciles Qdrant against the registry closes the never-re-ingested orphan window. Until then the gap is documented and bounded, not silently assumed away.
**Enforcement check (partial now, live later):** the orchestrator writes Qdrant before the registry (the ordering invariant is commented at the call site and asserted indirectly — `test_dedup.py` shows a registry-missing chunk classifies as upsert, i.e. self-heal); the never-re-ingested deletion-completeness case and the crash-mid-write self-heal are live checks under **ST-2** (real Postgres + Qdrant). The outbox/reconciliation itself is Phase-3 and unbuilt.

**Addendum (2026-06-30) — search-layer consistency: Qdrant is read-your-writes on a single node, eventually-fresh by ingestion cadence, and the only real eventual-consistency surface is multi-node.** A review raised the DDIA "search indexes build asynchronously → a query right after ingestion gets stale results" point. That is **Elasticsearch's** model (a `refresh_interval` before new docs are visible), **not Qdrant's**, and the distinction is the whole answer. Qdrant does not hide freshly-upserted points while indexing: a new point is immediately part of its segment, and until the segment crosses `indexing_threshold` (default 20k vectors) Qdrant **brute-force/exact-scans** the unindexed portion — so a just-ingested chunk is *immediately* searchable, in fact more accurately (exact, not approximate HNSW), only with worse latency. HNSW build lag is a **latency/recall** characteristic, never a **visibility** one. Where eventual consistency actually lives:
- **`wait` on upsert (single node):** `wait=false` would ACK before the write is applied — the one single-node staleness window. The qdrant-client default is **`wait=True`** (`qdrant_client.py`), so the ingestion writer is ACK'd only once the point is durable *and* searchable. **Decision: the Phase-3 Qdrant sink adapter keeps `wait=True`** — never trade it for write throughput without revisiting this (a batch writer can afford it).
- **Replication lag + read `consistency` (multi-node only):** the reader (`qdrant.py` `query_points`) sets no `consistency`, so a *clustered* deployment would read one possibly-lagging replica. **Decision (FUTURE, when Qdrant goes multi-node): set read `consistency` (e.g. `"majority"`) and write `ordering` deliberately** rather than inheriting "read one replica." Not relevant single-node.
- **No read-your-own-writes *coupling* exists by design:** ingestion is a decoupled batch/connector pipeline, not a synchronous upload→search flow, so freshness is bounded by **ingestion cadence**, not an index-refresh window. A just-landed doc being momentarily absent is the system's accepted nature (eventually fresh), not a bug — and `wait=True` means it appears the instant the writer's upsert returns.
**Enforcement check (partial now, future later):** the reader uses the client default (no `wait=false` write path exists yet); the sink adapter, when built, must pass `wait=True` and is the place the multi-node `consistency`/`ordering` choices land (`# FUTURE EXTENSION` on the build-plan sink-adapter row).

## DD-21 — Turn persistence idempotency needs an end-to-end key: the client supplies `message_id`, the server-minted UUID is only a fallback
**Fixes a real exactly-once gap (DDIA Ch. 11–12).** Turn persistence was *built* idempotent — `messages` is `INSERT … ON CONFLICT (message_id) DO NOTHING`, `turns` is `ON CONFLICT (turn_id) DO UPDATE` — and `Message`'s docstring called `message_id` "the idempotency key for persistence." But every id was a fresh server-side `uuid4()` minted inside `send_message.execute()` per request, and `ChatMessageRequest` carried no client key. **An idempotency key that isn't stable across the retry can't suppress the duplicate it exists for:** under at-least-once delivery (double-click, proxy/LB retry of the POST, dropped SSE → reconnect-and-resend) each attempt minted new ids, the `ON CONFLICT` never matched, and a second user+assistant turn was inserted (plus a re-billed LLM call). The dedup machinery was correct but keyed on the wrong thing.

**Decision:** the idempotency key is **end-to-end** — `ChatMessageRequest.message_id` (optional, client-generated, reused on retry) flows through the route into `execute(client_message_id=…)`, and all three persisted ids derive from it (`<id>` user message, `<id>:a` assistant, `<id>:t` turn) so a retry collides on every primary key. With no client key we fall back to a fresh `uuid4()` — today's at-least-once behavior, unchanged — so the fix is backward-compatible and opt-in by the client. This deduplicates *persistence*; it does **not** suppress the duplicate *work* (a retry still re-embeds/re-retrieves/re-calls the LLM unless the single-turn response cache hits) — short-circuiting duplicate work on an in-flight key is a `# FUTURE EXTENSION`.
**Enforcement check (real now):** `test_chat_security.py::test_client_message_id_makes_persisted_ids_stable_across_retry` (two executes with the same `client_message_id` persist identical `message_id`/`turn_id`, so ON CONFLICT dedups) and `::test_without_client_message_id_ids_differ_per_request` (fallback mints distinct ids — the documented at-least-once behavior).

## DD-22 — The ingestion worker's blocking-call rule is its own: sync clients via `asyncio.to_thread` are allowed, and the AV scanner fails **closed**
**Draws the line the Core API's "zero blocking on the event loop" invariant does *not* cross — and fixes the security posture of the first worker adapter (the clamd AV scanner).** Two decisions, set together because the clamd adapter is the *first* concrete adapter in the worker and establishes the pattern the rest of Phase-3 (Presidio, pypdf/python-docx, boto3) will follow.

- **Blocking rule (scoped, not global).** CLAUDE.md's async-everywhere / "zero blocking calls on the event loop" rule is a **Core API** (`src/`) invariant — it protects a latency-critical, always-on request path where one blocked coroutine stalls every concurrent request. The ingestion worker is a **throughput-oriented batch** deployable that ingests items **sequentially** (`ingest_batch`'s `async for`): there is no concurrent request path to starve. So the worker is free to reuse a mature **synchronous** client and push its blocking socket work off the loop with `asyncio.to_thread`, rather than hand-roll an async re-implementation of a wire protocol. **Decision: prefer well-tested sync libraries wrapped in `to_thread` for worker adapters; do not pay the complexity of a bespoke async client where the batch shape doesn't demand it.** (`adapters/av_scanner/clamd.py` wraps the sync `clamd` INSTREAM client this way.) This is a deliberate *narrowing* of the async invariant to where it earns its keep, not a repeal — the moment the worker grows intra-batch concurrency, per-adapter blocking budgets get revisited.
- **Fail-closed on scanner outage.** The AV scan can fail to produce a verdict (daemon unreachable, timeout, transport error, unparseable reply). That is categorically different from a *clean* verdict, and treating "couldn't scan" as "no virus" would make a scanner outage an open door — the exact inverse of the security goal. **Decision: the scanner raises `AvScannerUnavailable` (a port-level exception, not a `ScanResult`) on any non-verdict, and the security gate turns it into a `scan_error` quarantine** — the item is rejected and never parsed, matching the Prompt Guard fail-closed stance. The exception lives on the **port** (`ports/av_scanner.py`), not the adapter, so the gate (a pipeline stage) catches it without importing an adapter — preserving the hexagonal rule. Fail-*open* was rejected: a knowledge-ingestion path is precisely where an attacker would want the scanner "conveniently" down.
**Enforcement check (real now):** `test_av_scanner_clamd.py` — Tier 1 (mocked client, always runs): `FOUND`→infected+signature, `OK`→clean, and connection-error / `ERROR`-status / garbled-reply all raise `AvScannerUnavailable`, with `test_gate_quarantines_on_scanner_outage` asserting the gate maps that to `reason="scan_error"` (not a crash); Tier 2 (real `clamd` client over an in-process fake clamd TCP server, skipped if the lib is absent): clean/EICAR verdicts and daemon-down fail-closed over the **actual INSTREAM socket path**. Live verification against a real clamav daemon + real EICAR file is **ST-2**.

## DD-23 — `chunk_id` is a deterministic UUIDv5, not a sha256 hex, because it doubles as the Qdrant point id
**Fixes a latent bug in the frozen chunk-identity contract, caught the moment the first real Qdrant writer (the sink adapter) was built.** `contracts/chunk_identity.md` defined `chunk_id` — which is *also* the Qdrant point id — as `sha256(...).hexdigest()`, a 64-char hex string. But **Qdrant only accepts an unsigned 64-bit integer or a UUID as a point id**; a raw sha256 hex is rejected at upsert with `ValueError: Point id … is not a valid UUID` (reproduced against `qdrant-client`'s in-memory engine before writing a line of the adapter — the offline `:memory:` mode is exactly what surfaced it without a live daemon). The contract as written was therefore unimplementable; nothing had hit it because Phases 0–2 ran entirely on fakes.

**Decision: derive `chunk_id` as `uuid5(namespace, "{source}\x1f{native_id}\x1f{field_role}\x1f{seq}")`** over the *same* canonical tuple, with a fixed platform namespace (`uuid5(NAMESPACE_URL, "eaiplatform/contracts/chunk_id")`) so it is stable across processes/machines. UUIDv5 preserves every property the sha256 carried — deterministic (idempotent re-ingest overwrites the same point), no timestamps/randomness, and unit-separator collision-safety (`('ab','c') ≠ ('a','bc')`, since the name strings still differ) — while being a legal point id. This keeps the contract's **"point id == payload `chunk_id` == `chunk_id`" same-value invariant true**, so the core-api retriever (`chunk_id=str(point.id)`) needs **no change**, and the payload schema (which types `chunk_id` as a bare `string`, no pattern) is unaffected.

**Rejected alternatives:** (a) *keep sha256 as `chunk_id`, derive a separate UUID point id, make readers pull `chunk_id` from the payload* — introduces two identifiers, breaks the same-value invariant, forces a retriever change and a `uuid5()` mapping on every read/delete-by-id; strictly more surface for no benefit. (b) *first-8-bytes-of-sha256 as a uint64 id* — worse collision margin and still loses the same-value property. `content_hash` (delta/dedup) **stays a sha256 hexdigest** — it is a content fingerprint, never a point id, so Qdrant's id validation never touches it.
**Enforcement check (real now):** `test_identity.py::test_chunk_id_is_a_valid_uuid` (parses + round-trips as a `uuid.UUID`) plus the unchanged determinism / component-variation / separator-collision tests; `test_sink_qdrant.py` upserts against the in-memory Qdrant engine and asserts `point.id == chunk.chunk_id` and idempotent overwrite by that id. Both contract tests (worker + core-api) stay green. Live multi-node behavior is **ST-2**.
