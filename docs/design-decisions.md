> **SCOPE — CROSS-CUTTING.** This is the running design-decision log for the **whole project** (Core API + sidecars + ingestion worker + frontend). It records *why* we chose things, so we stay consistent across sessions and deployables. Newest decisions at the bottom. Per-deployable build plans/logs live in their own `docs/*-build-plan.md` / `*-dev-log.md`.

# Design Decisions (running log)

## DD-1 — Embedding sidecar runs on CPU, not GPU
The model server is **always-on** and serves **one short query per request** at interactive QPS — a *latency* workload, not a *throughput* one. GPU is only justified for bulk throughput, which belongs to the ingestion worker (off-hours, ephemeral GPU). Query sidecar = CPU + (later) ONNX int8. Cheap CPU replicas scale horizontally with QPS.

## DD-2 — bge-m3 backend: FlagEmbedding first, ONNX int8 later
Phase-1 uses FlagEmbedding's reference `BGEM3FlagModel` (PyTorch CPU) because it's correct (dense + learned-sparse guaranteed) and doubles as the **FP32 ground truth** to validate a future ONNX int8 export against. ONNX int8 is deferred until benchmarks prove it's needed. Local CPU bench (12-core box): intra_op=2 → p50 830ms; **intra_op=4 → p50 469ms (sweet spot)**; intra_op=12 → 581ms (regresses). Default set to `EMBED_INTRA_OP_THREADS=4`. Still above the ~250ms target → ONNX int8 is the next lever.

## DD-3 — Embedding/inference stays in Python (not Rust/Go)
The bottleneck is the native ONNX/torch forward pass — identical in any language. A Rust/Go rewrite only replaces the thin request wrapper (microseconds vs ~100s of ms inference) while forcing us to reimplement bge-m3's sparse post-processing. The GIL concern is moot: inference releases the GIL, so the thread-pool offload already gives true parallelism. Revisit only at very high QPS for p99.

## DD-4 — Messy user queries: rewrite-before-embed, not embed-as-is
bge-m3 handles typos/casing/punctuation/multilingual for free (do **not** lowercase/scrub — it's cased; over-cleaning strips signal). But it can't condense rambling or resolve cross-turn references ("why is *it* failing?"). So a **cheap LLM (Haiku) query-rewrite** turns the messy/multi-turn input into a clean **standalone** search query, and *that* gets embedded. Two distinct texts:
- **rewritten query → embedding/retrieval** (clean, intent-focused, context-resolved)
- **raw query → answer LLM** (preserve the user's actual words/tone)

Skip the rewrite when it can't help (first turn + already-clean/standalone query); only rewrite on multi-turn or long/rambling input.

## DD-5 — Two-tier response cache
- **L1 — raw-normalized key**, checked **first**, before any LLM call. Instant/free; catches exact repeats.
- **L2 — rewritten-query key**, checked only after L1 miss (rewrite already paid to embed). Catches different phrasings of the same intent; shareable across users with the same tenant + scope because the rewrite is standalone.

Pay the rewrite only on an L1 miss. Fill L2 only with the **standalone** rewritten form (never anything carrying session-specific references), or the next user's context gets a wrong answer.

## DD-6 — Agent semantic retrieval reuses the SAME embedding sidecar
Endpoint separation (chat vs agent) is about *orchestration flow*, not about avoiding embeddings. If an agent needs **semantic** Qdrant data it MUST embed via the same sidecar (`RetrieverPort`), exposed as a scope-checked agent **tool** that reuses the chat retrieval use-case — never a second embedding model. Only **exact/filter lookups** (by `tenant_id`/`permissions`/`ticket_id`/`doc_id`) skip embedding (Qdrant payload filter / Postgres). The real-time-status agent (1a) bypasses Qdrant entirely and hits live systems via MCP. **Not built yet — implement when agent tools land** (also noted in the agent flow section of `core-api-architecture.md`).

## DD-7 — Agent security: contain at the ACTION layer, not the input layer
**Thesis:** Prompt Guard (the sidecar) is only the **outer perimeter**. It catches blatant one-shot jailbreaks and misses what matters for an agent wired to prod systems (ServiceNow, GitHub, MCP tools): **indirect injection** (malicious instructions inside a ticket body / PR comment / retrieved doc / tool response) and **slow, gradual context poisoning** across turns. No input classifier can close that gap. So we **assume the agent's context will be poisoned** and move the trust boundary to the **action/authorization layer**. The question is never "is this prompt malicious?" but "is **this agent**, with **this scope**, allowed to take **this action** on **this system**, right now — and is it reversible?" The LLM **proposes**; deterministic code **decides**. Security guarantees live in the code *around* the model, so "the agent got talked into it" is never sufficient to cause harm.

**Layers (defense-in-depth):**
1. **Least-privilege scoped tools** *(foundation, partly built — `PermissionScope` gates MCP tool registration)*: per-task minimal tool set, read-only by default; writes to prod systems are a separate elevated class.
2. **Deterministic action-policy gate** *(key piece to add)*: an OPA/Cedar-style allow-list keyed on scope + action-class + target sits between the agent's *proposed* tool call and execution. A fully manipulated agent still hits the wall ("this scope may never delete prod resources"). Highest-leverage control.
3. **Human-in-the-loop for irreversible / high-blast-radius actions** *(already FUTURE in architecture)*: plan-then-apply; no merge/deploy/destructive change without explicit approval.
4. **Untrusted-data handling**: all tool/retrieved content is **data, never instructions**; structural separation of system vs user vs tool content; **provenance/taint** so high-taint external content can't authorize high-privilege actions. Second use of the guard: screen **tool outputs / retrieved docs** for indirect injection, not just the user prompt.
5. **Trajectory monitoring** *(for the slow attack)*: because the gate is at action-time, gradual poisoning still hits the policy wall when it finally acts; on top, an **independent monitor** watches the *sequence* of tool calls for drift toward sensitive ops + **cumulative session risk** (not per-message). Maps onto the Phoenix observability/drift/eval layer.
6. **Blast-radius containment (assume compromise)**: sandboxed code exec (E2B, FUTURE), **short-lived per-action scoped credentials** (agent never holds standing prod keys), rate limits / circuit breakers on sensitive tools, `agent_reaper` kills runaway/orphaned agents; prefer reversible, idempotent actions.
7. **Full audit**: every tool call + policy decision emits an OTel span to Phoenix (forensics + eval datasets).

**Built vs gap:** *Already designed* — `PermissionScope` top-down, MCP tool gating, scope-checked tool registration, human-in-the-loop (FUTURE), `agent_reaper`, observability/drift/evals, E2B sandbox (FUTURE). *To add explicitly* — the deterministic **action-policy engine**, **provenance/taint**, **tool-output guard screening**, **trajectory monitor + cumulative risk**, **per-action short-lived scoped credentials** + read/write tool separation + approval workflow. **Implement when agent tools land** (see DD-6 and the agent flow in `core-api-architecture.md`). → Pressure-tested and formalized into concrete, enforcement-checkable decisions in **DD-8…DD-12**.

## DD-8 — Policy Decision Point (PDP): a mandatory, agent-adversarial action chokepoint
**Refines DD-7.** Every agent-initiated effect on an external system (MCP tool call, sub-agent delegation, artifact-as-action) routes through a single deterministic Policy Decision Point *before* execution. The PDP is **default-deny** and must be correct under the assumption that **the agent is the attacker** — its decision may not depend on the model being honest, nor on taint (see DD-9). The LLM *proposes*; the PDP *decides*. "The agent was convinced" can never authorize an effect, because conviction lives in the data plane and authority lives in the control plane (DD-7), and they do not touch.

Pressure-tested amendments (vs. the naïve "allow/deny on action-class"):
- **Gate flows, not just actions.** Exfiltration is two individually-allowed actions (read in-scope source → write in-scope sink). The PDP evaluates `(scope, action-class, target, source-of-data, reversibility, quantitative bound)` — read-source × write-sink pairs are policy objects, not free. *Two-plane invariant, amended:* the model may move data **only within a policy-approved flow**.
- **Bounded parameters, not coarse classes.** `write:servicenow:ticket` cannot distinguish "close one ticket" from "close every P1." Prod-write tools MUST be individually modeled with typed, bounded parameters (max count / rate / blast-radius). This is an accepted, non-zero modeling cost — coarse classes are gameable on parameters.
- **Canonical targets.** The PDP resolves the real target itself (adapter-bound concrete resource); it never trusts a model-supplied label like `target=dev-db`.
- **Delegation attenuates.** The PDP sits at the execution boundary for *all* agents incl. sub-agents / A2A peers; scope may only *narrow* across delegation, never widen (closes the confused-deputy bypass — A2A is already stubbed in `src/adapters/agent/a2a/`).

**Why highest-leverage:** a fully poisoned agent still hits a deterministic wall it cannot argue with. This is the single control that converts "we have a security philosophy" into "an agent physically cannot mutate prod from poisoned context."
**Enforcement check (future):** no code path may invoke `MCPConnectorPort.call_tool` (or any write-capable tool/adapter) except via the PDP; a write tool with no registered bounded-parameter policy is default-denied; an architecture-style test asserts the chokepoint has no bypass. → **Implemented** (core, no caller yet): `core/use_cases/policy/policy_decision_point.py` (default-deny `decide()` — scope, attenuating delegation, canonical-target spoof check, flow gating, bounded params, reversibility/approval, taint-as-signal); `src/tests/test_pdp_chokepoint.py` (static no-bypass guard) + `test_policy_decision_point.py`. Still future: the credential broker (DD-10), trajectory monitor (DD-11), approval backend (DD-12), and the PDP-guarded MCP adapter that becomes its first caller.

## DD-9 — Taint/provenance is a SIGNAL, not a gate (you cannot trace influence through an LLM)
Every context fragment is tagged with source-trust (user / first-party retrieval / external tool output / external doc). **But taint must never be the thing a hard deny depends on.** Pressure test: you cannot prove whether tainted content *influenced* an LLM decision, and if you treat all model output as tainted (its context is tainted), taint becomes universal and the PDP denies everything → useless. Therefore taint tracks only **provable data-flow into action parameters**, and is consumed as: (a) a risk input to the trajectory monitor (DD-11), (b) friction/justification-raising on high-privilege actions, (c) audit/forensics. It is **defense-in-depth, not load-bearing.** DD-8 must be correct with taint unknown.
**Enforcement check (future):** removing the taint subsystem must not make any DD-8 deny become an allow (i.e., no hard policy rule keys on taint alone).

## DD-10 — Capability-based, short-lived per-action credentials (assume compromise)
The agent holds **no standing prod credentials.** To act, it requests a short-lived, single-purpose, scope-narrowed capability token minted per approved action. This caps blast radius when (not if) an agent is compromised, and makes every credential issuance an auditable PDP event. Prefer reversible, idempotent operations.
**Enforcement check (future):** no long-lived prod secret is reachable from the agent runtime; credential TTL ≤ action window; every mint is tied to a PDP allow decision.

## DD-11 — Trajectory monitor + cumulative session risk (the answer to the SLOW attack)
**Promoted from "hardening" to required.** Pressure test: the user's actual threat (slow context poisoning / gradual privilege elevation) is a chain of *individually-allowed* actions, which a stateless per-action PDP (DD-8) cannot catch. An **independent monitor** scores the *sequence* of tool calls and accumulates **session-level risk** (not per-message), watching for drift toward sensitive ops, read-then-exfiltrate shapes, and elevation gradients; crossing a threshold throttles, forces re-approval (DD-12), or kills the session (`agent_reaper`). Rides the Phoenix observability/eval layer. **DD-8 stops the single catastrophic action; DD-11 stops the slow accumulation — neither alone is sufficient.**
**Enforcement check (future):** a synthetic "thousand small cuts" trajectory must trip the monitor even though every individual action passes DD-8. → **Implemented** (core, no caller yet): `core/use_cases/policy/trajectory_monitor.py` (stateful per-session cumulative risk; signals: mutating-drift, read-then-exfiltrate, elevation-gradient, privilege-growth, probing; OK→THROTTLE→REQUIRE_APPROVAL→KILL). `test_trajectory_monitor.py::test_thousand_small_cuts_trips_monitor_though_each_passes_pdp` runs every action through the real PDP (asserting ALLOW) and shows the monitor still escalates to KILL. Still future: Redis-backed `SessionRiskStore` (in-memory now), Phoenix spans, and the agent-runtime caller that acts on the verdict (`agent_reaper`).

## DD-12 — Reversibility (not "sensitivity") is the axis for human-in-the-loop
Approval gating keys on **irreversibility / blast-radius**, not a vague "sensitive" label. Plan-then-apply: no merge/deploy/destructive/irreversible action executes without explicit human approval; reversible idempotent actions flow without friction. Reversibility is a first-class attribute of every tool's policy contract (consumed by DD-8 and DD-11).
**Enforcement check (future):** every write-capable tool declares a reversibility class; anything `irreversible` with no approval token is default-denied at the PDP.

### Pressure-test summary (why these five, in this shape)
Red-teamed before adoption. Survivors and corrections: **two-plane separation holds** but the invariant became "data moves only within an approved *flow*"; **DD-8 holds and hardened** (flows + bounded params + canonical targets + attenuating delegation, correct under an adversarial agent); **DD-9 demoted** (influence is unprovable through an LLM → taint is a signal, not a gate); **DD-11 promoted** (the per-action gate is blind to slow accumulation by construction). The honest one-liner: **DD-8 stops the single catastrophic action; DD-11 stops the slow accumulation; DD-10/DD-12 limit blast radius; DD-9 is a smoke detector, not a lock.** All five carry an *Enforcement check* so we can later audit designed-vs-enforced (cf. the architecture-boundary test precedent).

## DD-13 — Untrusted retrieved-content handling for RAG (chat path)
**Refines DD-9, applied to RAG reads.** Retrieved corpus chunks are **untrusted data, never instructions**. Indirect injection (a malicious instruction embedded in a corpus doc) is mitigated in layers; *scanning is the secondary layer, structure is the primary one.*

- **Layer 0 — Structural separation in the system prompt (primary, free, chat path).** Context is injected in a delimited, datamarked block; the system prompt explicitly declares it untrusted reference data — "quote it, never obey it." This is the actual gate against indirect injection and costs nothing per query. (Today `send_message.py` injects context as `[{i}] {text}` — delimited but lacking the "this is data, not commands" framing + anti-injection instruction; that framing is the highest-ROI hardening.)
- **Layer 1 — Scan at INGESTION, not retrieval (amortized).** The **ingestion worker** (`ingestion_worker/`, not yet built — this is a build requirement for it) screens each chunk **once at ingest**: cheap heuristics first (imperative 2nd-person directives, "system prompt"/"ignore previous" language, zero-width/invisible unicode, HTML comments, base64 blobs, embedded delimiters), then the Prompt Guard (PG2) classifier only on suspects. Result is persisted as **first-class fields on the chunk payload** (Qdrant), alongside `permissions`: e.g. `screened: bool`, `injection_risk: float`, `provenance/source-trust`. Cost is paid once per document, off the hot path — never once-per-query-per-popular-doc.
- **Layer 2 — At retrieval, consume the stored flag (cheap, no hot-path model call).** Query time reads the precomputed flag. A flagged chunk is **neutralized, not hard-dropped**: down-ranked, wrapped in explicit "⚠ untrusted/flagged" markers, taint raised. Hard-drop is itself a weapon — poisoning one sentence of a legit doc would erase it from retrieval (a DoS on knowledge). Hard blocking is reserved for the **agent** path (where a chunk could authorize an action, DD-8); in chat the blast radius is only "wrong answer," so over-blocking costs more than it protects.
- **Layer 3 — Optional batched retrieval-time classifier pass** for high-risk tenants only. Opt-in, not default.

**Design rules:** *ingest-time > retrieval-time* (scan once, amortize, keep the hot path clean); *neutralize/flag > block* for chat; the classifier score is a **signal feeding the taint flag, never an autonomous gate** (consistent with DD-9 — structure is the gate, the score is the smoke detector).
**Scope note (chat vs agent):** in chat (read-only, pre-scoped retrieval) a successful injection is bounded to *answer integrity / misinformation*, not breach — confidentiality/isolation already holds by construction. DD-13 closes the answer-integrity gap; it does not exist to prevent a breach (there isn't one to prevent on the chat path).
**Enforcement check (future):** every chunk written by the ingestion worker carries a `screened` flag + `injection_risk` (no chunk reaches Qdrant unscreened); the chat system prompt contains the untrusted-context framing; a synthetic poisoned chunk is down-ranked/marked at retrieval rather than silently used; removing Layer 1/3 must not change chat *confidentiality* (only answer integrity).
